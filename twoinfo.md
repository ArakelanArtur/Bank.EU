# Этап 4. Fullstack MVP

Срок: 5-7 дней

## Описание проекта

Нужно разработать fullstack MVP для учебного финансового сервиса `LumenBridge Finance Ltd`, который принимает онлайн-заявки на краткосрочные займы для физических лиц и малого бизнеса в Европе.

Исходный клиентский контекст находится в файле [`04-fullstack-client-content.md`](./04-fullstack-client-content.md). Используйте его как источник текста для понимания продукта, сценариев, разделов сайта и юридических ограничений. 

Проект должен реализовать все страницы и основные функции, описанные в клиентском ТЗ. Это не лендинг с одной формой, а полноценный учебный продукт с публичной частью, личным кабинетом пользователя и административной панелью.

Проект должен показать полный рабочий путь заявки:

- пользователь знакомится с сервисом на публичной части сайта;
- рассчитывает примерные условия займа;
- отправляет заявку;
- backend проверяет и сохраняет заявку;
- пользователь получает доступ к личному кабинету;
- пользователь отслеживает заявки, займы, платежи и уведомления;
- оператор управляет заявками, клиентами, займами, платежами и статусами в админ-панели;
- изменения в админ-панели отражаются в личном кабинете пользователя.

Это учебный MVP, а не production-банкинг. Реальные платежные интеграции, SMS-провайдеры, внешние скоринговые сервисы и документооборот заменяются локальными mock-сценариями. В интерфейсе и README должно быть понятно, какие части являются имитацией.

## Стек

Frontend:
- Next.js;
- TypeScript;
- Tailwind CSS v4;
- React Hook Form + Valibot;
- архитектура фронтенда Feature-Sliced Design.
Backend:
- Node.js;
- NestJS.

Хранение данных:
- PostgreSQL;

Дополнительно:

- клиентская и серверная валидация обязательны;
- frontend и backend должны быть связаны через HTTP;
- URL backend должен быть вынесен в конфигурацию или переменную окружения;
- CORS должен быть настроен, если части проекта работают на разных портах.
- личный кабинет и админ-панель должны работать с сохраненными данными backend, а не с изолированными mock-массивами в компонентах.
- вход пользователя должен быть реализован через имитацию SMS OTP;
- вход администратора должен быть реализован по логину и паролю;
- в админ-панели должна быть ролевая модель с ролями `admin` и `operator`.

## Definition of Done

Проект считается завершенным, если выполнены все обязательные пункты.

Страницы и разделы:

- реализована главная страница со всеми смысловыми блоками из ТЗ;
- реализована страница "Как это работает";
- реализована страница для бизнес-клиентов;
- реализована страница FAQ;
- реализованы контакты и форма обратной связи;
- реализована политика конфиденциальности;
- реализована cookie policy;
- реализованы остальные юридические страницы или разделы, если они указаны в ТЗ;
- реализован личный кабинет пользователя;
- реализована административная панель;
- интерфейс адаптивен и не ломается на мобильном экране.

Калькулятор:

- использует формулу аннуитетного расчета:
  `A = P × (r × (1 + r)^n) / ((1 + r)^n − 1)`;
- где `A` — размер платежа, `P` — сумма займа, `r` — процентная ставка в день, `n` — срок займа в днях;
- дневная ставка для расчета в учебном проекте: `0.8% = 0.008`;
- общая сумма к возврату рассчитывается по формуле:
  `Total = A × n`;
- пользователь видит размер платежа и общую сумму к возврату;

Форма заявки:

- собирает данные, достаточные для первичной заявки;
- поддерживает физическое лицо и бизнес-заявителя;
- создает или связывает пользователя с заявкой;
- проверяет обязательные поля на frontend;
- показывает пользователю ошибки валидации;
- показывает loading-состояние при отправке;
- показывает успешный результат с номером или идентификатором заявки;
- показывает понятную ошибку, если backend отклонил запрос.

Полноценный flow заявки:

- новая заявка появляется в админ-панели;
- оператор может взять заявку в обработку;
- оператор может одобрить или отклонить заявку;
- при одобрении создается займ в статусе ожидания подписания пользователем;
- пользователь видит займ, ожидающий подписания, в личном кабинете;
- подписание займа происходит через OTP-код;
- при подписании backend сохраняет дату подписания, IP-адрес и user agent пользователя, подписавшего документ;
- после успешного подписания график платежей создается автоматически;
- займ отображается пользователю в личном кабинете;
- по займу отображаются сумма, срок, ставка или расчетные условия, график платежей и текущий статус;
- пользователь может оставить заявку на оплату, указав сумму платежа и реквизиты или reference выполненного перевода;
- заявка на оплату попадает в админ-панель и ожидает ручной проверки;
- администратор или оператор вручную проверяет заявку на оплату и помечает ее выполненной или отклоненной;
- администратор фиксирует платежи по займу вручную;
- если зафиксированный платеж превышает сумму ближайшего платежа по графику, график пересчитывается и сдвигается соответствующим образом;
- после подтверждения платежа данные о нем и обновленный график отображаются в личном кабинете пользователя;
- оператор может отмечать платежи и менять состояние займа;
- закрытые займы переходят в историю пользователя;
- пользователь получает уведомления о ключевых изменениях статусов;
- все изменения сохраняются и остаются после обновления страницы.

Backend:

- запускается локально отдельной командой;
- принимает заявки с frontend;
- повторно валидирует данные на сервере;
- сохраняет заявки в выбранное хранилище;
- хранит пользователей, заявки, займы, платежи и уведомления;
- хранит заявки на оплату и их статусы;
- хранит факт подписания займа, дату подписания, IP-адрес и user agent подписанта;
- автоматически создает график платежей после подписания займа;
- поддерживает ручное подтверждение платежей администратором;
- пересчитывает график платежей после подтверждения платежа, если внесенная сумма меняет ближайшие обязательства по графику;
- возвращает понятные ответы об успехе и ошибках;
- обеспечивает данные для публичной формы, личного кабинета и админ-панели;
- поддерживает изменение статусов и обновление связанных сущностей;
- не хранит реальные секреты и реальные персональные данные.

Личный кабинет пользователя:

- пользователь входит по номеру телефона через имитацию SMS OTP;
- OTP-код не отправляется через реальный SMS-провайдер, но flow запроса и подтверждения кода должен быть реализован в интерфейсе и backend;
- личный кабинет создается автоматически при первой заявке;
- есть навигация по разделам;
- есть раздел заявок со статусами;
- есть раздел активных и закрытых займов;
- есть детальная карточка займа;
- есть сценарий подписания займа через OTP-код;
- есть график платежей;
- есть действие подачи новой заявки из кабинета;
- есть действие просмотра договора как mock-документа или заглушки;
- есть действие создания заявки на оплату с указанием суммы и реквизитов платежа;
- пользователь видит статус своей заявки на оплату;
- есть раздел уведомлений;
- есть состояния загрузки, пустого результата и ошибки.

Административная панель:

- есть вход в админ-панель по логину и паролю;
- есть роли `admin` и `operator`;
- `admin` может управлять всеми сущностями и учетными записями админ-панели;
- `operator` может обрабатывать заявки, клиентов, займы, платежи и уведомления в пределах рабочего сценария, но не управляет ролями и учетными записями админов;
- есть навигация по разделам;
- оператор видит список заявок из backend;
- оператор может искать и фильтровать заявки;
- оператор может открыть карточку заявки;
- оператор может изменить статус заявки, одобрить или отклонить ее;
- оператор может оставить комментарий к заявке;
- есть раздел клиентов;
- оператор может искать и фильтровать клиентов;
- оператор может открыть карточку клиента;
- карточка клиента показывает контактные данные, историю заявок, активные займы и историю платежей;
- есть раздел займов;
- оператор может видеть активные и закрытые займы;
- оператор может открыть карточку займа;
- оператор может изменить статус займа, отметить платеж и закрыть займ;
- есть раздел платежей;
- оператор или администратор видит заявки на оплату от пользователей;
- оператор или администратор может вручную проверить заявку на оплату;
- оператор или администратор может подтвердить или отклонить заявку на оплату;
- администратор вручную фиксирует поступившие платежи;
- после фиксации платежа система обновляет график платежей и связанные статусы;
- оператор может подтверждать платежи и отмечать просрочки;
- есть раздел системных уведомлений;
- интерфейс показывает loading, empty и error states;
- изменения в админ-панели сохраняются и отражаются в личном кабинете пользователя.

## Ограничения

Не расширяйте scope за счет реальных финансовых интеграций и production-безопасности. Главная цель задания - показать полный локальный fullstack flow, валидацию, хранение, интеграцию frontend/backend, личный кабинет, админское управление сущностями и способность довести проект до демонстрационного состояния.
